# Release v0.4.7d-plan-etapa4.7d-plan - Contrato API y decision backend cPanel

## Estado general

ETAPA 4.7D-PLAN es una etapa documental. Define el contrato minimo de API, los DTO esperados y la decision tecnica recomendada para backend en cPanel antes de implementar integraciones reales.

No se implemento codigo funcional. No se implemento backend, MySQL, API routes, autenticacion real, roles reales ni migraciones. La app sigue funcionando como demo/MVP con `localStorage` como repositorio activo por defecto.

## Contrato API definido

El contrato API propuesto cubre los endpoints minimos futuros para:

- Autenticacion: `login`, `logout`, `me`.
- Bodegas.
- Trabajadores.
- Inventario.
- Consumos.
- Reportes operativos.
- Auditoria.

El contrato queda alineado al MVP operativo: bodega, inventario, trabajadores/invitados, consumo de EPP, vale/ficha imprimible, reportes operativos y Excel.

## Compatibilidad con MVP

`projectId` no vuelve a ser obligatorio. El contrato propone mantenerlo como referencia historica opcional mediante `projectIdLegacy`, sin reactivar proyectos como eje del MVP.

`requesterReference` queda como texto opcional para centro de costo, faena o area solicitante. No se convierte en maestro nuevo.

Los costos pueden existir internamente para trazabilidad, pero no quedan definidos como foco visual del vale ni de reportes operativos.

## Decision backend recomendada

Para cPanel tradicional, la recomendacion principal es:

- Frontend Next.js exportable/estatico.
- API separada PHP/Laravel.
- MySQL/MariaDB del hosting.

Node/Express + MySQL queda como alternativa solo si el proveedor confirma soporte real de Node.js, Passenger/Application Manager, SSH, variables de entorno, logs persistentes y politica clara de procesos.

Next.js completo con servidor Node en cPanel no se recomienda como primera opcion si el proveedor no garantiza soporte estable. Vercel/Render/Railway son alternativas tecnicamente mejores para Node, pero se alejan del requisito comercial de hosting cPanel.

## Estado de repositorios de datos

`localStorageRepository` sigue siendo el comportamiento por defecto de la app.

`apiRepository` permanece como preparacion/stub para etapas futuras y no queda activo por defecto.

No se cambio `useData`, no se cambio `localStorageRepository`, no se cambio `storage-keys` y no se modifico el flujo de stock.

## No implementado

En esta etapa no se implemento:

- Backend real.
- Conexion MySQL.
- API routes.
- PHP/Laravel.
- Node/Express.
- Autenticacion real.
- Roles reales.
- Middleware server-side.
- Migracion de datos.
- Variables `.env`.
- Dependencias nuevas.

## Riesgos pendientes

- El proveedor cPanel podria no ofrecer Node.js real o estable.
- MySQL compartido podria tener limites de rendimiento.
- La migracion desde `localStorage` podria contener datos inconsistentes.
- La autenticacion real aun no existe.
- La proteccion de rutas aun no es server-side.
- La concurrencia real de stock todavia no esta implementada.
- El alcance podria crecer hacia ERP, finanzas, aprobaciones o roles complejos.

## Siguientes etapas recomendadas

- ETAPA 4.7D-BACKEND-SETUP: crear estructura inicial del backend elegido, sin conectar aun produccion.
- ETAPA 4.7E: adaptar `apiRepository` contra endpoints reales en ambiente controlado.
- ETAPA 4.8: autenticacion real y seguridad minima piloto.
- ETAPA 4.9: pruebas reales con usuarios y datos de empresa.
