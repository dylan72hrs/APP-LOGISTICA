## 2. Arquitectura recomendada

Recomendación principal para el MVP:

- Frontend Next.js estático + API separada + MySQL.

Criterios:

- Si cPanel no garantiza Node.js estable, preferir API PHP/Laravel + MySQL por compatibilidad con hosting tradicional.
- Node/Express + MySQL solo debe considerarse si el hosting confirma Node.js real, Passenger/Application Manager, acceso SSH, variables de entorno y logs utilizables.
- Vercel/Render/Railway quedan como alternativa técnica válida para Node/Next, pero se alejan del requisito comercial de cPanel.
- No se recomienda migrar directo desde `localStorage` a backend productivo sin una etapa intermedia de repository/API y pruebas.

## 4. Migración desde localStorage

Estrategia propuesta:

- Exportar respaldo local JSON desde la clave `app-logistica:persistent-data:v1`.
- Validar estructura del JSON antes de importar.
- Importar bodegas.
- Importar trabajadores.
- Importar inventario.
- Importar consumos.
- Crear detalle de consumos desde cada item registrado.
- Comparar conteos entre origen local y MySQL.
- Validar stock por bodega/producto después de importar.
- Registrar importación inicial en `audit_log`.
- Mantener respaldo original sin modificar.
- Activar `apiRepository` solo después de pruebas y validación de datos.

- No cambiar la app completa de golpe.

## 7. Seguridad futura

La seguridad real queda para etapas posteriores, pero debe diseñarse con:

- `password_hash` para contraseñas.
- Cookies `HttpOnly`, `Secure` y `SameSite`.
- Middleware server-side para rutas protegidas.
- Roles mínimos: `admin`, `operator`, `reports`.
- Protección real de `/admin/users`.
- Rate limiting en login.
- HTTPS obligatorio.
- Auditoría de consumo e inventario.
- No guardar tokens sensibles en `localStorage`.
- No depender de ocultar botones como mecanismo de seguridad.

## 8. Concurrencia de stock

El consumo con descuento de stock debe ejecutarse en transacción:

- Iniciar transacción.
- Bloquear productos con `SELECT FOR UPDATE` o mecanismo equivalente.
- Validar bodega.
- Validar trabajador.
- Validar que cada producto pertenece a la bodega.
- Validar stock suficiente dentro de la transacción.
- Insertar consumo.
- Insertar items del consumo.
- Descontar inventario.
- Registrar `audit_log`.
- Hacer `COMMIT` si todo es válido.
- Hacer `ROLLBACK` ante cualquier error.

Este flujo evita doble descuento cuando dos operadores consumen el mismo EPP al mismo tiempo.

## 10. Riesgos

- cPanel sin Node real.
- MySQL compartido lento.
- Migración desde `localStorage` con datos inconsistentes.
- Auth mal implementada.
- Pérdida de datos por importación sin respaldo.
- Doble descuento de stock sin transacciones.
- Alcance creciendo hacia ERP, finanzas o roles complejos.
- Logs y backups insuficientes del proveedor.
- Deploy manual difícil de repetir.
