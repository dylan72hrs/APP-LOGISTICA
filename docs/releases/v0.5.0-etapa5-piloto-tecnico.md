# Release v0.5.0-etapa5-piloto-tecnico - Preparacion de piloto tecnico

## Estado

ETAPA 5 prepara un piloto tecnico controlado para validar backend PHP, MySQL
staging y frontend en modo API.

No es produccion. No activa API por defecto. No implementa auth real, roles
avanzados, aprobaciones, integracion Jury, multi-pais ni IA.

## Que se preparo

- Checklist operativo para cPanel, MySQL staging, backend, frontend y rollback.
- Plan de pruebas staging con casos positivos y negativos.
- Plantilla de resultados para registrar evidencia del piloto.
- Seed SQL ficticio minimo para smoke test en staging.
- Script CLI `check-schema.php` para validar tablas y columnas criticas sin
  exponer credenciales.

## Que se valido localmente

- `npm.cmd run typecheck`: OK.
- `npm.cmd run build`: OK.
- PHP lint: no ejecutado en este entorno porque `php` no esta disponible.

Las pruebas reales contra DB staging quedan pendientes si no existe configuracion
staging segura disponible.

## Seguridad

- No se agregaron credenciales.
- No se creo `.env` real.
- No se creo `DatabaseConfig.php` real.
- No se agregaron host, usuario ni password reales.
- No se agregaron dumps reales ni datos productivos.
- El seed contiene datos ficticios de staging.

## Repositorio de datos frontend

`localStorage` sigue siendo el modo default del frontend.

El modo API sigue siendo opt-in y solo debe usarse con:

```text
NEXT_PUBLIC_DATA_REPOSITORY=api
NEXT_PUBLIC_API_BASE_URL=<backend-staging-url>
```

Al terminar un smoke test, se debe quitar el flag o volver a:

```text
NEXT_PUBLIC_DATA_REPOSITORY=local
```

## Proyectos en piloto

Proyectos ya forma parte del flujo operativo del piloto. El consumo debe probarse
con proyecto activo, codigo, centro de costo y dimension financiera.

`POST /consumptions` debe probarse con:

- `projectId` obligatorio.
- stock suficiente.
- stock insuficiente.
- producto de otra bodega.
- proyecto inactivo.
- rollback.
- generacion de `audit_log`.

## Pendiente en cPanel/staging

- Confirmar version PHP real del hosting.
- Confirmar PDO MySQL habilitado.
- Crear DB staging.
- Crear usuario DB staging con permisos minimos.
- Cargar schema.
- Cargar seed ficticio o datos staging aprobados.
- Configurar DB por entorno o archivo local no versionado.
- Ejecutar `check-db.php`.
- Ejecutar `check-schema.php`.
- Probar endpoints GET.
- Probar `POST /consumptions`.
- Probar frontend con API.
- Volver a `localStorage` al terminar.

## Criterios para declarar piloto aprobado

- Typecheck OK.
- Build OK.
- PHP lint OK.
- `GET /health` OK.
- DB staging conectada sin exponer secretos.
- Schema requerido completo.
- GET de proyectos, bodegas, trabajadores e inventario OK.
- `POST /consumptions` OK con proyecto y stock suficiente.
- `POST /consumptions` rechaza stock insuficiente sin descuento parcial.
- Rollback verificado.
- `audit_log` verificado.
- Frontend API smoke OK.
- Frontend vuelve a `localStorage`.
- Sin credenciales ni datos reales versionados.

## Riesgos pendientes

- No hay auth real.
- No hay roles/permisos reales.
- No hay proteccion productiva de rutas.
- No hay gestion de sesiones real.
- cPanel puede limitar logs, variables de entorno, permisos o conexiones.
- No hay backup/restauracion verificada en staging.
- No hay pruebas de concurrencia reales fuera de staging.
- `apiRepository` sigue siendo smoke controlado, no integracion final de produccion.
- CORS debe endurecerse con origen especifico antes de cualquier uso con datos reales.
