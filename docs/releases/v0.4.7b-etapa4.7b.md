# Release v0.4.7b-etapa4.7b - Preparar apiRepository y feature flag sin activar backend

## Estado general

ETAPA 4.7B completa como preparacion minima de arquitectura para futura migracion desde `localStorage` hacia API/backend.

Esta etapa no activa backend real y mantiene la app funcionando por defecto con `localStorage`.

## Cambios aplicados

- Se agrego `src/lib/data/api-repository.ts` como stub preparado.
- Se agrego `src/lib/data/repository-factory.ts` para seleccionar repository por feature flag publica futura.
- Se conecto `useData` al factory sin cambiar su contrato publico.
- `localStorageRepository` sigue siendo el repository default.
- La estructura persistida y la clave `app-logistica:persistent-data:v1` no cambiaron.

## Feature flag preparado

Variable futura:

```text
NEXT_PUBLIC_DATA_REPOSITORY=local|api
```

Comportamiento:

- Sin variable: usa `localStorageRepository`.
- `local`: usa `localStorageRepository`.
- `api`: selecciona `apiRepository`, pero este aun es un stub sin endpoints reales.
- Valor invalido: usa `localStorageRepository` y emite advertencia tecnica controlada.

## apiRepository

`apiRepository` implementa la interfaz actual `DataRepository`:

- `load(seed)`
- `save(snapshot)`

Estado actual:

- No llama endpoints reales.
- No crea API routes.
- No persiste en backend.
- Devuelve seed en `load`.
- No-op en `save`.
- Emite advertencia explicita indicando que aun no esta implementado.

## Riesgos pendientes

- La app sigue usando `localStorage` por defecto.
- `apiRepository` no esta listo para piloto real.
- La interfaz `DataRepository` sigue siendo snapshot/sincrona; una API real probablemente requerira contrato async por entidad.
- No hay backend, MySQL, auth real, auditoria real ni concurrencia server-side.
- La migracion real queda para etapas posteriores.
