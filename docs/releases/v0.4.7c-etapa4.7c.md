## Qué se creó

### Esquema SQL base

El archivo `docs/database/schema-mysql.sql` define las tablas mínimas para el MVP futuro:

- `warehouses`
- `users`
- `workers`
- `inventory_items`
- `consumption_records`
- `consumption_items`
- `audit_log`

El esquema usa:

- Motor `InnoDB`.
- Charset `utf8mb4`.
- Llaves primarias.
- Llaves foráneas.
- Índices útiles.
- `created_at` y `updated_at` donde corresponde.
- `active` para soft delete lógico donde corresponde.
- UUID documentado como `CHAR(36)`.

No incluye:

- Datos reales.
- Correos reales.
- Passwords reales.
- Credenciales demo.
- Seeds productivos.

### README de base de datos

El archivo `docs/database/README.md` explica:

- Propósito del esquema.
- Tablas incluidas.
- Orden recomendado de importación:
  1. `warehouses`
  2. `users`
  3. `workers`
  4. `inventory_items`
  5. `consumption_records`
  6. `consumption_items`
  7. `audit_log`
- Relación con `app-logistica:persistent-data:v1`.
- Que el esquema todavía no está conectado a la app.
- Que la migración real queda para etapa posterior.
- Riesgos de ejecutar SQL sin respaldo.
- Recomendación de probar primero en base staging.

### Checklist cPanel

El archivo `docs/deployment/cpanel-checklist.md` incluye preguntas y verificaciones sobre:

- Soporte Node.js.
- Versión Node.js.
- Passenger/Application Manager.
- SSH.
- Variables de entorno.
- MySQL/MariaDB y versión.
- phpMyAdmin.
- Backups automáticos.
- Restauración puntual.
- SSL.
- Límites CPU/RAM/procesos.
- Logs.
- Cron jobs.
- Subdominio API.
- Política ante procesos Node colgados.
- Permisos para conexión local a MySQL.
- Capacidad de staging.

También documenta la recomendación:

- Si cPanel no garantiza Node.js estable, usar frontend estático + API PHP/Laravel + MySQL.
- Si cPanel sí soporta Node estable, evaluar Node/Express o Next.js server con cuidado.

## Riesgos pendientes

- El esquema aún no ha sido probado en un cPanel real.
- La versión MySQL/MariaDB del proveedor puede requerir ajustes menores.
- `JSON` en `audit_log` depende del soporte del motor disponible.
- La migración desde `localStorage` puede traer datos inconsistentes.
- Sin backend no existe concurrencia server-side de stock.
- Sin auth real no hay seguridad productiva.
- Ejecutar SQL en una base con datos reales sin respaldo puede causar pérdida de datos.
